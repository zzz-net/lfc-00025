# 实验室传感器质控分析看板

> Sensor Quality Control Dashboard - 温湿度传感器异常检测与人工复核工作台

## 功能概览

| 模块 | 说明 |
|------|------|
| **数据导入** | 一键导入样例数据 / 拖放上传 CSV/JSON，自动校验时间戳、数值合法性 |
| **多设备管理** | 左侧卡片式传感器列表，点击切换查看不同设备数据 |
| **双轴图表** | 温度/湿度双 Y 轴折线图，阈值虚线参考线，异常点散点叠加标记 |
| **异常检测** | 7 类异常：温度越界（上/下限）、湿度越界、温漂、湿漂、数据断点 |
| **阈值配置** | 8 个阈值参数可调整，修改后自动重算，**已人工标注记录永久保护** |
| **人工标注** | 异常状态流转：待处理 → 待确认 / 已接受 / 误报 / 需复测 |
| **回滚机制** | 一键回滚最近一次标注（软删除，保留完整历史审计） |
| **报告导出** | CSV 双表 + PDF 横向 A4，包含概览、阈值、异常明细、标注历史 |
| **持久化** | SQLite 本地存储，重启后筛选条件、图表、标注、报告全部保留 |

## ✅ 标准验证流程（必跑）

以下步骤按顺序执行，可完整验证「筛选保存 → 导出一致性 → 回滚 → 跨重启 → 再导出」整条链路。

### 前置：启动并导入数据

1. 运行 `npm run dev`，浏览器打开 http://localhost:5173
2. 点击顶栏 **「导入样例」** 按钮，等待提示「样例导入成功」
3. 左侧传感器列表出现 5 台设备

### 步骤 1：设置筛选条件（单个传感器 + 只看已接受异常）

1. 左侧点击 **「SN-1001 冷库A-1号」**，等待图表加载
2. 右侧异常面板顶部，点击 **「已接受」** 状态徽章（绿色）
3. 顶栏时间范围切换为 **「7D」**（或者「自定义」选 2025-05-01 ~ 2025-05-07）
4. 观察右侧异常面板计数：**共 X 条 · 筛选 Y 条**，记下 Y 的数值

### 步骤 2：标注几条异常为「已接受」

1. 在右侧异常列表中，找到任意 2~3 条待处理异常，点击卡片展开
2. 点击 **「人工标注」** 按钮
3. 状态选 **「已接受」**，处理人填 `testuser`，原因写 `验证流程：人工确认异常`
4. 提交后卡片变为绿色受保护状态，顶部「已接受」计数增加
5. 重复标注 2~3 条

### 步骤 3：导出 CSV，验证内容与界面一致

1. 确认当前筛选：传感器=SN-1001、状态=已接受、时间=7D（筛选计数=刚才标注的条数）
2. 点击顶栏 **「CSV报告」** 按钮，浏览器下载 `.csv` 文件
3. 用 Excel 或文本编辑器打开 CSV：
   - ✅ **只出现 SN-1001 传感器**，不混进 SN-1002~1005
   - ✅ **状态列全部是「已接受」**，没有「待处理」「待确认」等
   - ✅ **条数 = 界面筛选计数 Y**
   - ✅ 处理人列显示 `testuser`，原因列显示刚才填写的文字

### 步骤 4：回滚一次标注

1. 在右侧异常面板，点击 **「历史」Tab** → 最新一条标注右上角「回滚此条」
   - 或者 Tab 右侧 **↶ 回滚最近一次** 按钮 → 填回滚原因 `验证流程：误操作回滚` → 确认
2. 回滚后：
   - ✅ 该异常状态恢复为 **「待处理」**（红色）
   - ✅ 顶部「已接受」计数减 1
   - ✅ 「历史」Tab 中该条出现「已回滚」灰色标签，并显示回滚原因

### 步骤 5：修改阈值触发重算，验证人工标注不被冲掉

1. 顶栏点击 **「阈值配置」**
2. 把温度上限从 30 调整到 **32**，点击「保存并重新计算」
3. 弹窗提示：`阈值已更新，重算异常完成：新增 N 条，保护人工结论 X 条`
   - ✅ **保护数 = 你之前标注且未回滚的条数**
4. 回到异常列表，检查之前标注过且未回滚的那几条：
   - ✅ 依然是「已接受」绿色状态
   - ✅ 处理人、原因、标注时间完整保留
   - ✅ 回滚掉的那一条保持「待处理」

### 步骤 6：跨重启验证筛选持久化

1. **完全关闭** 运行 `npm run dev` 的终端（Ctrl+C 两次）
2. 重新执行 `npm run dev`，浏览器刷新页面
3. 验证：
   - ✅ 左侧传感器列表依然显示 5 台（数据没丢）
   - ✅ **默认选中 SN-1001**（重启后 selectedSensorId 恢复）
   - ✅ 右侧状态徽章 **「已接受」** 被选中（statusFilter 恢复）
   - ✅ 顶栏时间范围 **7D**（timeRange 恢复）
   - ✅ 之前标注的处理人、原因、回滚记录全部存在

### 步骤 7：重启后再导出一份报告，验证结果不漂

1. 重启后不要动任何筛选条件
2. 再次点击 **「CSV报告」** 下载第二份 CSV
3. 与步骤 3 下载的第一份对比：
   - ✅ 传感器、状态、时间范围完全一致
   - ✅ 条数完全相同
   - ✅ 处理人、原因、回滚信息完全相同（仅 CSV 文件名和生成时间有差异）

### 步骤 8：重复导入同一批数据，验证无重复 & 标注不丢

1. 再次点击 **「导入样例」**（导入完全相同的一批数据）
2. 提示：`数据已存在，重复导入被拒绝` → 证明 file_hash 去重生效
3. 左侧传感器读数计数 **没有翻倍**
4. 之前所有人工标注、回滚、状态保持不变

|-----------|------|------|------|---------|

| 行号 | 传感器 | 类型 | 说明 |
|------|--------|------|------|

## ⚙️ 配置入口

### 1. 阈值配置

**入口**：顶栏 → 「阈值配置」按钮，或快捷键菜单

| 分类 | 参数 | 默认值 | 单位 | 含义 |
|------|------|--------|------|------|
| **温度** | tempMin / tempMax | 15 / 30 | ℃ | 温度上下限，越界立即标异常 |
| **湿度** | humidMin / humidMax | 30 / 70 | %RH | 湿度上下限 |
| **漂移** | tempDriftThreshold | 2 | ℃ | 相邻读数温度差 > 此值判定为漂移 |
| | humidDriftThreshold | 10 | %RH | 相邻读数湿度差阈值 |
| **断点** | gapThresholdSeconds | 600 | 秒 | 相邻读数时间差超过判定数据断点 |

> 💡 **人工保护机制**：修改阈值点击「保存并重新计算」后，系统会 **只删除并重建未标注的异常**。已被质控人员手动标注的异常（无论状态是已接受/误报/待确认）都带有 `has_manual_override=1` 标志，**永远不会被重算覆盖**。

### 2. 导入配置（隐式）

| 配置项 | 说明 |
|--------|------|
| 文件格式 | `.csv` / `.json` / `.txt`，最大 20MB |
| CSV 表头识别 | 中英文自动映射：`传感器/设备编号/sensor_id`、`时间/记录时间/timestamp`、`温度/temperature`、`湿度/humidity` |
| 错误处理 | 时间戳格式错误、读数非数字、空值 → 跳过并累计到错误列表，不中断导入 |
| 重复导入 | 对文件内容做 SHA256 哈希，`import_batches.file_hash` UNIQUE 约束，完全相同文件二次导入会被拒绝 |

## 🔍 复核步骤（标准操作流程 SOP）

### 第一步：导入数据

1. 登录看板后点击 **「导入样例」**（或 **「上传文件」** 选择自己的 CSV）
2. 上传模式下会先显示 **实时校验结果**：
   - ✅ 校验通过 → 行数/传感器数展示
   - ⚠️ 警告 → 行号 + 说明（通常是异常值提示）
   - ❌ 错误 → 脏数据行，会被导入时跳过
3. 点击「开始导入」，完成后左侧传感器列表自动刷新

### 第二步：切换传感器、查看图表

1. **左侧卡片**点击任意传感器（推荐先点 `SN-1001 冷库A-1号`，异常最集中）
2. **图表时间切换**：顶部 1H / 24H / 7D / ALL / 自定义（选择起止时间）
3. **图表识别**：
   - 🟠 橙色折线 = 温度（左 Y 轴）
   - 🔵 青色折线 = 湿度（右 Y 轴）
   - 虚线 = 阈值上下限
   - 彩色圆点/菱形/方形 = 异常点，鼠标悬停显示异常类型列表

### 第三步：人工标注异常

1. **右侧异常面板**查看当前传感器的异常列表
2. 用顶部 **6 类状态筛选徽章**（全部/待处理/待确认/已接受/误报/需复测）快速过滤
3. 点击卡片 **展开详情** → 查看读数、检测算法参数快照、历史标注记录
4. 点击 **「标注处理」** 按钮（或直接点击异常卡片）弹出右侧标注面板：
   - 选择 **状态分类**：待确认 / 已接受 / 误报 / 需复测（每项带图标和说明）
   - 填写 **处理人**（自动记忆上次 localStorage）
   - 填写 **原因 / 处理说明**（最多 500 字，实时计数）
5. 点击提交按钮（颜色跟随状态变化）→ 状态变为带边框的受保护状态

### 第四步：回滚一次标注

1. 标注后发现选错了？有两种方式回滚：
   - 异常面板顶部 **「历史」Tab** → 最新标注卡片右上角「回滚此条」
   - 异常面板 Tab 右侧 **↶ 回滚最近一次** 按钮 → 弹窗填写回滚原因 → 确认
2. 回滚后：
   - 该条异常状态恢复为 `待处理 (DETECTED)`
   - 如果这是该异常最后一条标注，保护标志 `has_manual_override` 自动解除
   - 下次重算阈值时该异常会被算法重新生成
   - 回滚记录保留在操作历史中，审计可查

### 第五步：导出质控报告

点击顶栏右侧两个按钮：

| 按钮 | 格式 | 内容 |
|------|------|------|
| **CSV报告** | `.zip` 含 2 个 CSV | Sheet1：异常明细（类型、状态、处理人、原因）；Sheet2：标注历史（含回滚）|
| **导出PDF** | 横向 A4 `.pdf` | 多页：①概览（传感器/异常/标注统计） ②阈值配置表 ③异常明细表 ④标注历史表（状态列彩色渲染）|

## 🏗️ 技术架构

```
前端 (React 18 + TypeScript)
├── 状态管理     Zustand (store/index.ts)
├── 图表         Recharts (双Y轴折线+散点)
├── UI组件       9 个 (TopNav / SensorList / SensorChart / AnomalyPanel ...)
└── 样式         TailwindCSS 3 + 自定义主题 slateqc/accent

后端 (Express 4 + better-sqlite3)
├── 路由层        api/routes/*.ts (7个模块)
├── 服务层        ImportService / AnomalyDetector / ReportService / SampleDataGenerator
├── 数据访问层    repositories/*.ts (6个Repo)
└── 工具层        csvParser (中英文表头) / fileHash (SHA256去重)

存储层
├── SQLite        api/data/qc_database.db (自动创建)
└── 7 张表：sensors / readings / import_batches / anomalies
              / annotations / thresholds / app_state
```

**关键表设计要点**：

| 表 | 关键字段 | 用途 |
|---|---------|------|
| anomalies | `has_manual_override INT` | 0=算法可覆盖 / 1=人工标注过永久保护 |
| annotations | `rolled_back_at DATETIME NULL` | 软删除标记，回滚填时间戳，不物理删除 |
| app_state | `key UNIQUE` | 单行存储 selectedSensorId / statusFilter / timeRange 等 UI 筛选条件 |

## 📋 异常类型字典

| Code | 中文名 | 触发条件 |
|------|--------|----------|
| `OVER_LIMIT_TEMP` | 温度超上限 | `temperature > tempMax` |
| `UNDER_LIMIT_TEMP` | 温度低于下限 | `temperature < tempMin` |
| `OVER_LIMIT_HUMID` | 湿度超上限 | `humidity > humidMax` |
| `UNDER_LIMIT_HUMID` | 湿度低于下限 | `humidity < humidMin` |
| `DRIFT_TEMP` | 温度漂移 | `|curr.temp - prev.temp| > tempDriftThreshold` |
| `DRIFT_HUMID` | 湿度漂移 | `|curr.humid - prev.humid| > humidDriftThreshold` |
| `DATA_GAP` | 数据断点 | `curr.ts - prev.ts > gapThresholdSeconds` |

## 📋 标注状态流转

```
DETECTED ──标注──► PENDING     待确认（需进一步核实）
           │
           ├────► ACCEPTED    已接受（确认为真实异常）
           │
           ├────► FALSE_POSITIVE  误报（系统误判）
           │
           └────► RETEST      需复测（安排重新测量）

任何状态 ──「回滚最近一次」──► DETECTED （软删除最后一条标注）
```

## ❓ 常见问题

**Q: 修改阈值后我的人工标注不见了？**
A: 不会。带人工标注的异常有保护标志，重算时不会删除。可能是筛选条件切换了，请检查右侧面板顶部的状态筛选徽章。

**Q: 同一批数据导了两次会怎样？**
A: 文件内容哈希完全一致会被拒绝导入。如果文件改了哪怕一行（例如修改了文件名），会导入成功但读数会去重（按 sensor_id + timestamp 唯一）。

**Q: 回滚能回滚几次？**
A: 「回滚最近一次」按钮只回滚最新的一条标注，但可以连续点。历史 Tab 中可以看到所有回滚痕迹。

**Q: 数据库在哪里？怎么清空？**
A: 数据库文件：`api/data/qc_database.db`。完全清空数据请停止服务后删除该文件，下次启动自动重建空表。
